# 前端图片匹配卡片与相关文章分组设计

## 背景

当前静态首页已经具备上传图片、展示人物信息、展示图片匹配结果和相关文章的能力，但前端还有三个和结果表达相关的问题：

1. 左侧存在用户可见的状态卡片，会把接口内部状态直接暴露给前端用户。
2. 图片匹配卡片当前同时展示缩略图、标题、来源和相似度，视觉重心不够集中，且卡片高度对不同图片比例适配不足。
3. 相关文章仍按单列表展示，无法和图片匹配结果建立清晰的一一对应关系。

本次改动目标是收敛前端展示逻辑，让结果页由“图片匹配结果”驱动相关文章组织方式，同时不扩大后端接口变更范围。

## 目标

- 取消左侧用户可见状态卡片 `statusBox`，不再单独展示成功、失败、部分完成等状态文案。
- 保留接口 `status` 字段给前端脚本内部判断，不调整后端响应结构。
- 让图片匹配卡片更适配结果图片，突出图片本身，弱化文字信息。
- 将图片匹配结果中的 `title` 从卡片上移除，改为放入“相关文章”区域作为每组文章的标题线索。
- 相关文章区域改为按 `image_matches` 的顺序逐组展示，和图片卡片顺序保持一一对应。
- 每个文章组都需要表明对应图片结果的来源信息。

## 非目标

- 不修改后端接口字段定义。
- 不新增后端“图片结果到文章结果”的显式映射模型。
- 不移除调试面板中的原始 JSON，也不隐藏其中的 `status` 字段。
- 不调整人物信息卡和社交账号卡的数据结构。

## 方案比较

### 方案一：仅在前端重组展示，保持接口结构不变

前端继续接收现有的 `image_matches` 和 `news` 数据，在静态页内部重构渲染方式：

- 匹配卡片只显示缩略图、来源、序号和相似度。
- `title` 从卡片移到相关文章区域。
- 相关文章按图片匹配顺序渲染为多个文章组。

优点：

- 改动集中在静态页，风险最小。
- 不触碰响应模型，对现有后端逻辑影响最小。
- 能最快满足当前页面交互需求。

缺点：

- 前端只能按现有数据能力做展示分组，图片与文章的关联语义不是后端显式建模。

### 方案二：后端补充显式分组结构

后端新增“图片匹配结果对应文章组”的响应结构，前端直接消费该结构。

优点：

- 语义最明确，后续扩展最稳定。

缺点：

- 需要调整对外响应结构，与当前尽量收敛前端改动的目标不符。
- 变更范围更大，需要同步补充更多接口层测试。

### 方案三：只隐藏卡片标题，不改文章区域结构

只从匹配卡片里移除 `title`，相关文章仍保留单列表。

优点：

- 改动最小。

缺点：

- 无法满足“排序按照图片排序一一对应”的核心需求。

## 推荐方案

采用方案一，只调整静态页展示逻辑，不改后端响应模型。

原因：

- 用户需求明确聚焦在前端结果呈现而非接口能力。
- 现有接口已经提供足够的图片卡片数据，前端可以在不改模型的前提下完成大部分重排。
- 当前仓库已有静态页前端渲染逻辑，继续沿用该模式可控性最高。

## 页面结构设计

### 1. 左侧上传区

- 保留上传表单、图片预览和提交按钮。
- 删除用户可见状态卡片 `statusBox` 的 DOM 结构和相关样式。
- 删除对 `statusText`、`statusNote`、`statusBox` 的直接界面更新逻辑。
- 前端仍保留接口 `status` 的内部判断，用于错误分支和空态文案控制，但不再渲染独立状态模块。

### 2. 图片匹配卡片

图片匹配区域继续作为结果页的重要入口，但改为更强调图片内容：

- 卡片主体突出缩略图。
- 卡片信息区只保留：
  - 匹配序号
  - 来源 `source`
  - 相似度 `similarity_score`
- 不在卡片中显示 `title`。
- 卡片布局需要允许不同纵横比的图片较完整展示，优先使用 `object-fit: contain` 和更灵活的图片区高度。
- 点击卡片仍然跳转到对应来源链接。

### 3. 相关文章区域

相关文章区域从“单列表”改为“按图片分组”：

- 以 `image_matches` 顺序作为顶层分组顺序。
- 每个分组对应一张图片匹配结果。
- 分组标题使用该图片原有的 `title`。
- 分组标题附近展示来源 `source`，明确该组文章对应的图片结果来源。
- 分组内部继续渲染文章列表，单篇文章沿用现有字段：
  - `title`
  - `source`
  - `published_at`
  - `summary`
  - `url`

### 4. 空态与失败态

- 当没有图片匹配结果时，图片区域维持现有空态，但文案应聚焦“暂无图片匹配结果”。
- 当存在图片匹配结果但没有可展示文章时，相关文章区域不再只显示一个总空态，而是允许：
  - 渲染分组容器
  - 在分组内显示“该图片结果暂无相关文章”
- 当接口失败时，保留现有错误卡片处理思路，但不再借助左侧状态卡片提示。

## 数据流与前端逻辑设计

### 输入数据

继续沿用当前接口响应：

- `status`
- `error`
- `person`
- `image_matches`
- `news`

### 渲染逻辑

1. 提交后先重置人物卡、图片匹配区、相关文章区和社交账号区。
2. 请求成功后：
   - `renderPerson()` 继续负责人物卡。
   - `renderImageMatches()` 负责渲染无标题的图片卡片。
   - `renderArticleGroups()` 改为以 `image_matches` 为主输入，按顺序生成文章组。
3. 请求失败时：
   - 继续使用当前错误渲染逻辑更新主要结果区。
   - 不再调用任何用户可见状态卡片更新。

### 一一对应规则

本次不新增后端显式映射，因此页面的一一对应定义为：

- 文章组的排列顺序与 `image_matches` 顺序一致。
- 每个文章组的标题和来源信息来自该图片匹配项本身。
- 组内文章内容使用当前可用 `news` 数据填充。
- 如果现有数据无法稳定推导哪篇文章只属于哪张图片，则优先保证分组顺序、标题和来源信息正确，不伪造错误映射；没有可确认内容时显示组内空态。

这是一种“展示对应关系”而非“数据实体绑定关系”。如果后续需要精确绑定，应在后端单独建模。

## 样式设计

- 删除 `.status`、`.status-success`、`.status-partial`、`.status-failed` 及其相关文本样式的页面占位职责。
- 图片卡片需要调整为更适配缩略图的网格布局，避免现在固定高度导致信息区和图片区比例失衡。
- 相关文章分组需要新增组容器样式，和单篇文章卡片形成明确层级：
  - 组头展示标题和来源
  - 组内文章卡片保持现有浅色块风格
- 保持现有页面视觉基调，不引入新的整体视觉方向。

## 测试与验证

本次优先覆盖以下行为：

1. 无图片匹配结果时，图片区域和相关文章区域都能正确显示空态。
2. 有多张图片匹配结果时，相关文章区域的分组顺序与图片顺序一致。
3. 图片卡片不再展示 `title`，相关文章区域展示分组标题和来源。
4. 接口失败时，不再依赖左侧状态卡片提示，主体区域仍能正确展示错误结果。

验证方式：

- 先修改静态页并人工验证主要渲染路径。
- 若已有静态页相关测试，则补充最小覆盖。
- 最终运行 `mvn clean verify`，确认本次前端改动未影响后端构建与测试。

## 风险与后续

### 风险

- 由于后端没有显式文章分组字段，前端的一一对应是展示层约定，不是数据层强约束。
- 如果未来文章来源更加复杂，仅靠前端重排可能不足以准确表达关联关系。

### 后续可选增强

- 若后续需要真正的图片到文章精确映射，可在后端增加显式文章分组模型，再由前端直接渲染。
- 若图片来源标题质量不稳定，可再补一层标题清洗策略，但不属于本次范围。
