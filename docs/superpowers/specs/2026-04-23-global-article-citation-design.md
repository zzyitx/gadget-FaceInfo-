# 全局文章来源编号与摘要引用设计

## 1. 背景

当前人物聚合链路已经具备以下能力：

- 通过 `JinaReader` 抓取网页正文并生成 `PageSummary`
- 通过 `DeepSeek` / `Kimi` 生成最终人物总结与专题摘要
- 通过 `ParagraphSummaryItem.sources` / `sourceUrls` 为段落补充来源信息
- 前端结果页可以把正文中的 `[n]` 渲染为脚注链接

但现状仍有两个问题：

1. 单篇正文摘要、最终人物总结、各专题摘要之间没有共用一套稳定的文章编号。
2. 大模型虽可返回 `sources/sourceUrls`，但没有被明确约束为“每句推断必须显式标注来自哪篇文章”。

本次设计的目标，是把“编号”从前端展示层的临时脚注，升级为后端统一定义的“全局文章编号”，并要求三层摘要都显式引用这套编号。

## 2. 目标

### 2.1 功能目标

- 为参与摘要链路的文章建立全局唯一编号 `1..N`
- 单篇正文摘要、最终人物总结、各专题摘要共用同一套文章编号
- 大模型在输出每句总结时，必须在句尾追加来源编号，如 `[1]`、`[2][5]`
- 文章来源区按同一顺序输出完整文章清单

### 2.2 非目标

- 不调整主题扩展词推断逻辑
- 不改变已有聚合主流程的 provider 路由策略
- 不要求把“来源精确到正文原句片段”，本次只精确到“文章级别”

## 3. 核心规则

### 3.1 编号语义

- 编号本质上代表“文章编号”，不是段落编号，也不是脚注渲染时的临时序号
- 同一篇文章在整次人物结果中只能占用一个编号
- 同一篇文章无论出现在单篇正文摘要、最终人物总结还是专题摘要中，引用编号都必须一致

### 3.2 编号范围

- 编号作用域是“单次人物聚合结果”
- 对某次聚合进入摘要链路的全部文章，统一建立 `1..N`
- 不允许单篇正文摘要单独从 `1` 开始重新编号
- 不允许专题摘要单独从 `1` 开始重新编号

### 3.3 句尾引用格式

- 每句摘要后必须带至少一个编号
- 单来源示例：`他长期担任某企业核心管理者。[3]`
- 多来源示例：`公开资料显示其职业轨迹集中在金山、小米与投资布局。[2][4][7]`
- 多个编号按升序输出，去重后拼接
- 不允许输出未在文章表中出现的编号

### 3.4 文章来源区格式

- 文章来源区必须按编号顺序完整列出文章
- 每条至少包含：
  - `id`
  - `title`
  - `url`
  - `source`
  - `publishedAt`
- 展示顺序严格等于编号顺序

## 4. 方案选择

本次采用“后端统一建立全局文章索引表，再将其注入三层摘要 prompt”的方案。

### 选择原因

- 能从提示词阶段约束模型把每句推断和文章编号绑定
- 能保证三层摘要共用同一套编号，不依赖前端临时归并
- 能与现有 `sources/sourceUrls` 结构兼容，保留展示元数据

### 不采用的方案

- 不采用“前端按 URL 临时编号”的方案，因为编号不稳定，且无法对模型输出施加约束
- 不采用“仅后处理 URL 到编号映射”的方案，因为它不能保证模型在总结阶段已经显式声明依据

## 5. 数据模型调整

## 5.1 新增全局文章引用模型

建议新增独立模型，例如 `ArticleCitation`，字段如下：

- `id`: 全局文章编号，从 `1` 开始
- `title`: 文章标题
- `url`: 文章 URL
- `source`: 来源站点
- `publishedAt`: 发布时间

用途：

- 作为单次聚合结果的统一文章表
- 作为大模型 prompt 中可引用的文章索引
- 作为前端文章来源区的唯一数据源

## 5.2 现有摘要模型补充字段

### `PageSummary`

新增或补充：

- `sourceId`: 当前篇级摘要对应的全局文章编号
- `summaryParagraphs`: 单篇正文摘要段落列表
- 每个段落保留：
  - `text`
  - `sourceIds`
  - `sourceUrls`
  - `sources`

说明：

- 单篇正文摘要原则上只引用当前文章自身编号，但仍保留列表结构，避免后续兼容成本

### `ParagraphSummaryItem`

建议新增：

- `sourceIds: List<Integer>`

保留已有：

- `sourceUrls`
- `sources`

优先级规则：

- 后端标准化后以 `sourceIds` 作为主字段
- `sourceUrls/sources` 继续承担兼容与展示用途

### `SectionSummaryItem`

建议新增：

- `sourceIds: List<Integer>`

保留已有：

- `sourceUrls`
- `sources`

说明：

- 专题摘要虽然当前是“分段摘要”，但同样需要显式承载文章编号

### 聚合结果 / 最终人物画像

在最终返回结果中补充统一文章表，例如：

- `articleSources: List<ArticleCitation>`

该字段供前端文章来源区直接使用，不再依赖从各段落动态收集。

## 6. 编号生成规则

### 6.1 生成时机

全局文章编号在 `PageSummary` 集合稳定之后生成。

稳定点定义：

- 页面抓取完成
- 篇级摘要完成
- `mergePageSummaries` / 去重逻辑执行完成

### 6.2 去重规则

优先按 `sourceUrl` 去重：

- `sourceUrl` 相同，视为同一篇文章
- 首次出现的条目保留编号
- 后续重复条目复用相同编号

若 `sourceUrl` 缺失：

- 可回退到 `title + source` 组合键
- 若回退键也缺失，则不参与编号，视为不可信来源

### 6.3 顺序规则

编号顺序采用“摘要链路中的最终文章顺序”：

1. 先保留 `collectPageSummaries` / 合并后的顺序
2. 再按该顺序分配 `1..N`

这样可以保证：

- 排序稳定
- 与现有篇级摘要处理顺序一致
- 同一请求内编号可重现

## 7. Prompt 设计

三层摘要调用都必须接收同一份“文章索引表”上下文。

### 7.1 单篇正文摘要 prompt

输入：

- 当前文章正文
- 当前文章的全局编号
- 全局文章表（至少供模型理解编号语义）

约束：

- 只允许基于当前文章正文总结
- 每句末尾必须追加当前文章编号 `[n]`
- 不允许引用其他文章编号
- 必须返回结构化 JSON

目标输出示意：

```json
{
  "sourceId": 3,
  "summaryParagraphs": [
    {
      "text": "该文章显示其主要身份为企业家与投资人。[3]",
      "sourceIds": [3]
    }
  ]
}
```

### 7.2 最终人物总结 prompt

输入：

- 全部 `PageSummary`
- 全局文章表 `1..N`

约束：

- 只能依据提供的篇级摘要归纳
- 每句话必须以 `[n]` 或 `[n][m]` 结尾
- 引用编号只能来自文章表
- 若证据不足，不得编造结论

### 7.3 各专题摘要 prompt

输入：

- 对应专题相关的 `PageSummary`
- 同一套全局文章表

约束：

- 每句主题摘要必须带文章编号
- 若只来自 1 篇文章，只能引用该文章编号
- 若综合多个篇级摘要，可输出多个编号

### 7.4 统一提示词要求

三类 prompt 都必须包含以下硬约束：

- “编号代表文章编号，不代表段落编号”
- “每个句子后都必须给出来源编号”
- “禁止输出文章表中不存在的编号”
- “禁止在正文外解释编号规则”
- “只能返回约定 JSON，不要附加说明文本”

## 8. 服务层处理

## 8.1 统一文章表装配

服务层新增统一步骤：

1. 对参与摘要的 `PageSummary` 建立全局文章表
2. 为每个 `PageSummary` 回填 `sourceId`
3. 把文章表传入：
   - 单篇正文摘要调用
   - 最终人物总结调用
   - 各专题摘要调用

## 8.2 响应标准化

后端解析模型返回后，统一做标准化：

- 如果模型返回 `sourceIds`，直接校验并使用
- 如果模型只返回 `sourceUrls/sources`，后端按全局文章表映射补齐 `sourceIds`
- 如果模型正文中存在 `[n]`，编号必须与 `sourceIds` 一致

## 8.3 容错与降级

以下情况视为不合规：

- 句子缺少编号
- 编号不存在于文章表
- `sourceIds` 与正文中的 `[n]` 不一致
- URL 无法映射到全局编号

处理策略：

1. 先走现有重试
2. 仍失败时切换 provider 兜底
3. 若兜底仍不合规，则按当前错误体系返回摘要失败

## 9. 前端展示

## 9.1 摘要正文

前端继续复用现有 `[n]` 脚注渲染能力，但渲染数据源优先改为 `sourceIds`。

规则：

- 正文中的 `[n]` 继续渲染为可点击脚注
- 鼠标悬浮或点击时，从统一文章表中查找对应文章信息

## 9.2 文章来源区

前端文章来源区改为直接读取后端返回的 `articleSources`。

不再作为主路径依赖：

- `collectParagraphSourceArticles(person)` 这类从段落动态收集来源的逻辑

保留该逻辑只可作为兼容兜底，不作为正式来源顺序定义。

## 10. 测试设计

### 10.1 客户端测试

需要新增或调整：

- 单篇正文摘要请求体包含文章编号与全局文章表
- 最终人物总结请求体包含全局文章表
- 各专题摘要请求体包含全局文章表
- 解析 `sourceIds` 成功
- 仅返回 `sourceUrls/sources` 时可映射为 `sourceIds`
- 非法编号触发受控异常

### 10.2 服务层测试

需要覆盖：

- 同一文章在三层摘要中复用相同编号
- 合并后文章顺序稳定分配 `1..N`
- 重复 URL 不重复分配编号
- 文章来源区输出顺序等于全局文章表顺序

### 10.3 前端测试

需要覆盖：

- 摘要中的 `[n]` 能正确渲染
- 文章来源区按 `1..N` 顺序展示
- 同一编号在主体摘要、专题摘要中指向同一文章

## 11. 兼容性与风险

### 11.1 兼容性

- 保留 `sourceUrls/sources`，避免旧响应消费方立即失效
- 新增 `sourceIds` 与 `articleSources` 后，前端可逐步切换

### 11.2 主要风险

- 模型可能只输出正文中的 `[n]`，但不返回结构化 `sourceIds`
- 不同 provider 对“每句必须带编号”的遵循度可能不同
- 老测试大量基于 `summary/sourceUrls`，改造时容易遗漏

### 11.3 风险控制

- 在 JSON schema 中显式加入 `sourceIds`
- 在服务层做严格校验和补齐
- 保持 `sourceUrls/sources` 兼容，避免一次性切断旧逻辑

## 12. 实施边界

本次设计只覆盖以下范围：

- 全局文章编号生成
- 三层摘要 prompt 引用规则
- 响应结构补充 `sourceIds` / `articleSources`
- 前端文章来源区切换到统一文章表

不包含：

- 新的检索源接入
- 正文片段级证据定位
- 专题搜索策略重写

## 13. 结论

本设计将摘要链路中的“来源”从松散的 URL/来源对象，提升为统一的“全局文章编号体系”。

最终结果应满足：

- 单篇正文摘要、最终人物总结、各专题摘要共用同一套文章编号
- 每句推断都能明确标注依据文章
- 文章来源区顺序稳定且与摘要编号严格一致

这是后续实现“可解释摘要”和“可核对推断依据”的基础。
